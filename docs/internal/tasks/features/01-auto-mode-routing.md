# Фича: Автомодель (Auto/Autoselect) — Управление и Бенчмарки

## Статус и цель документа

Документ фиксирует согласованное решение по развитию автомодели (`autoselect`/`auto`) без лишнего усложнения.

Цель: сделать предсказуемый и управляемый `Auto`-режим, где:
- тип задачи определяет отдельный классификатор;
- целевая модель по типу задачи выбирается из настроек админки;
- есть бенчмарки для выбора классификатора и текстовых моделей;
- переключение победителей происходит только после подтверждения администратора.

## Термины

- `Автомодель` (`Auto`): пользовательский режим выбора модели `autoselect`/`auto`.
- `Классификатор`: модель, которая определяет тип задачи.
- Типы задач: `GENERAL`, `CODING`, `REASONING`, `VISION`, `IMAGE_GENERATION`.
- `GENERAL champion`: основная general-модель по результатам бенчмарка/подтверждения.

## Что делаем (функциональное решение)

1. Переименование в UI
- В `GPTHub Settings` переименовать раздел `Авторежим` в `Автомодель`.
- Сохранить двуязычный лейбл для пользовательского режима модели: `Авто` / `Auto`.

2. Маршрут запроса в Auto-режиме
- Пользователь выбирает `Auto` и отправляет сообщение.
- Запрос попадает в `gpthub-api`.
- `gpthub-api` вызывает модель-классификатор (настраивается в админке вручную).
- Классификатор возвращает: `label` + `confidence` + короткую `reason`.
- По `label` выбирается целевая модель из админских настроек по категориям задач.
- Запрос уходит в выбранную целевую модель, ответ возвращается пользователю.

3. Константы классификации (в админке)
- В разделе `Автомодель` добавить отдельную выпадающую категорию `Константы`.
- Внутри: переменные для классификатора (минимально):
  - `temperature`
  - `max_tokens`
  - `timeout_seconds`
  - `confidence_threshold`
  - `system_prompt_template`
  - `user_prompt_template`
- Значения используются рантаймом `gpthub-api` при вызове классификатора.

4. Поведение при ошибках/неуверенности
- Если классификатор не ответил, вернул невалидный формат или timeout:
  - fallback на определение типа задачи по keyword-правилам (RU/EN, расширенный словарь фраз в разных стилях).
  - если keyword-определение тоже неуверенное/неоднозначное, fallback в `GENERAL champion`.
- Если `confidence < confidence_threshold`:
  - fallback в `GENERAL champion`.
- Keyword fallback используется только как резервный путь после ошибки классификатора (не основной механизм).

5. Настройка целевых моделей по типам задач
- В админке в `Автомодель` добавить блок маппинга:
  - `GENERAL -> <model_id>`
  - `CODING -> <model_id>`
  - `REASONING -> <model_id>`
  - `VISION -> <model_id>`
  - `IMAGE_GENERATION -> <model_id>`
- Выбор вручную из выпадающих списков доступных моделей.

6. Бенчмарк классификатора
- Датасет: набор вопросов разных стилей + эталонный `label`.
- На входе бенчмарка: список general-моделей-кандидатов.
- Каждая модель-кандидат проходит весь датасет как классификатор.
- Метрики: минимум accuracy + macro-F1.
- Победитель не применяется автоматически: админ нажимает `Установить`.

7. Бенчмарк текстовых моделей (`GENERAL`, `CODING`, `REASONING`)
- Для каждой категории свой набор запросов.
- Кандидаты в категории сравниваются попарно турнирной сеткой.
- Судья (judge) выбирает лучший ответ в паре.
- Сетка строится по раундам до финального победителя.
- Победитель категории применяется только после ручного подтверждения `Установить`.

8. Визуализация турнирной сетки
- В админке добавить визуализацию сетки.
- Узлы сетки: прямоугольники с названиями моделей.
- Отрисовать раунды и переход победителей между раундами.

9. Еженедельный бенчмарк
- Переключатель в админке: `Еженедельный бенчмарк` (on/off).
- При включении выполняется плановый прогон:
  - классификатор-бенчмарк;
  - турнирные бенчмарки текстовых категорий.
- Результаты сохраняются в историю; применение изменений по-прежнему через ручное `Установить`.

10. Автоматическое определение профиля моделей (без доверия к хардкоду)
- Не полагаться на ручные/исторические пометки категорий моделей как на единственный источник истины.
- Добавить автоматический `capability probe` для новых/измененных моделей:
  - text chat probe;
  - vision probe;
  - image generation probe;
  - (опционально) audio/transcription probe.
- Добавить профильный benchmark по категориям задач (`GENERAL`, `CODING`, `REASONING`, `VISION`, `IMAGE_GENERATION`) и формировать категорию модели по фактическим результатам.
- Сохранять вычисленный профиль модели и время последнего пересчета.

11. Аватарки моделей (единый источник в `gpthub-api`)
- `gpthub-api` должен отдавать список всех моделей сразу с полем аватарки модели.
- Если у модели нет своей аватарки, подставляется стандартная аватарка из настроек `gpthub-api` (хранится на стороне `gpthub-api`).
- `openwebui` должен подтягивать и показывать аватарки напрямую из ответа `gpthub-api` без локального хардкода.
- В `GPTHub Настройки -> Общее` добавить возможность изменить стандартную аватарку моделей.

## Ограничения (что сознательно не делаем)

- Не внедряем сложную оркестрацию multi-step в этой задаче.
- Не включаем дополнительные схемы снижения смещения (blind/swap A/B) в рамках текущего объема.
- Не переводим систему на полностью автоматическое применение победителей без подтверждения админа.

## Критерий готовности

- Админ может полностью управлять автомоделью из одного раздела `Автомодель`.
- Классификатор и целевые модели меняются без правки кода.
- Бенчмарки запускаются, считаются метрики и доступны результаты.
- Турнирная сетка отображается визуально с названиями моделей.
- Победители применяются только после ручного подтверждения администратора.
