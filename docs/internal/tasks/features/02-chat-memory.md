# Фича: История и Долговременная Память Чата

## Цель

Сохранить полезный контекст между сообщениями и сессиями: предпочтения пользователя, факты, рабочие артефакты, ссылки на прошлые решения.

## Почему это важно для оценки

- Это обязательная фича задания.
- Прямо влияет на критерий "Качество автоматизации и памяти".
- Показывает, что продукт полезен в долгом цикле работы, а не только в одном сообщении.

## Как фича связана с общим сценарием

Пользователь возвращается в чат позже -> система поднимает релевантный контекст -> ответ учитывает прошлые решения и предпочтения -> пользователь видит и контролирует память.

## Где работать

- Backend:
  - Сервис памяти (сохранение, извлечение, обновление, удаление фактов).
  - Политика релевантности и ограничения на объем контекста.
  - Механизм связывания памяти с конкретными чатами/пространствами.
- Frontend:
  - История сообщений и поиск по истории.
  - Управление памятью: просмотр, редактирование, удаление, закрепление.
  - Маркеры "ответ построен с учетом памяти".
- Data/Storage:
  - Хранилище памяти, история изменений, срок жизни записей.

## План действий

- [ ] Определить типы памяти (профиль, проектные факты, временные заметки).
- [ ] Определить правила извлечения и обновления памяти.
- [ ] Добавить пользовательский контроль над сохраненными фактами.
- [ ] Встроить память в pre-processing каждого запроса.
- [ ] Настроить защиту от устаревшего/шумного контекста.
- [ ] Протестировать межсессионные сценарии и регрессии.
- [ ] Подготовить демо "до/после с памятью".

## Риски и зависимости

- Риск: формальная память без пользы (есть записи, но не влияет на ответы).
- Риск: устаревшие факты ухудшают качество.
- Зависимость: единая модель сообщений и метаданных.

## Критерий готовности для демо

- Система использует факты из прошлых диалогов осмысленно.
- Пользователь может посмотреть и удалить отдельные элементы памяти.
- Память улучшает ответ минимум в одном реальном сценарии.
