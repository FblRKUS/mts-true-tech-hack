# Фича: Единая Мультимодальная Модель Сообщений

## Цель

Сделать единый и последовательный UX для текста, голоса, фото и файлов: отправка, отображение, редактирование, версия сообщения, связь с агентом и памятью.

## Почему это важно для оценки

- Убирает эффект "разных режимов в разных местах".
- Повышает баллы за целостность UX/UI.
- Является базой для авторежима, памяти, OCR и превью.

## Как фича связана с общим сценарием

Пользователь отправляет голос/фото/файл -> сообщение сохраняется как единый объект -> агент получает структурированный контекст -> пользователь может редактировать текстовую расшифровку и видеть версии/ветки.

## Где работать

- Backend:
  - Единая схема `message` и `attachment`.
  - Версионирование сообщений и ветвление при редактировании.
  - Пайплайн STT с сохранением оригинального голосового сообщения.
- Frontend:
  - Композер с мультимодальными вложениями.
  - Плеер голосовых сообщений + текст расшифровки.
  - Маркеры "изменено после расшифровки", навигация по веткам.
- Интеграции:
  - Связка STT/OCR/инструментов с единым объектом сообщения.

## План действий

- [ ] Определить единый контракт мультимодального сообщения.
- [ ] Развести "оригинальный медиа-объект" и "редактируемый текст".
- [ ] Реализовать версионность и ветвление при редактировании.
- [ ] Добавить консистентное отображение вложений в ленте чата.
- [ ] Связать новые сообщения с памятью и автороутингом.
- [ ] Проверить UX на мобильном и десктопе.

## Риски и зависимости

- Риск: сложная модель данных может замедлить MVP.
- Риск: рассинхрон голоса и текста после ручного редактирования.
- Зависимость: стабильный storage для вложений и метаданных.

## Критерий готовности для демо

- Пользователь отправляет голос, слушает его, видит расшифровку и может ее исправить.
- Изменение текста не теряет исходный медиа-контент.
- Ветка редактирования сообщения отображается и не ломает диалог.
