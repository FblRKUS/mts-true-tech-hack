# Фича: Автопилот Выполнения Задач

## Цель

Позволить системе выполнять задачу итеративно: план -> выполнение -> самопроверка -> доработка до достижения критерия качества или лимита.

## Почему это важно для оценки

- Усиливает автоматизацию для сложных задач.
- Может дать качественный прорыв в задачах "сделай полностью".
- Демонстрирует зрелый агентный сценарий.

## Как фича связана с общим сценарием

Пользователь формулирует цель -> включает автопилот -> система делает несколько внутренних шагов -> показывает прогресс и итог -> при необходимости предлагает ручное вмешательство.

## Где работать

- Backend:
  - Оркестратор шагов автопилота.
  - Контроль лимитов итераций, токенов и времени.
  - Политика остановки и критерии качества.
- Frontend:
  - UI статуса этапов и прогресса.
  - Управление автопилотом: старт, пауза, остановка.
  - Просмотр итоговых артефактов.
- Безопасность:
  - Ограничения на рискованные действия.

## Текущее состояние (runtime политика)

- Автопилот не завершает задачу при одном только «код запускается» — требуется и технический, и продуктовый проход QA.
- Для backend/script-game задач supervisor не отправляет задачу в frontend/design ветки.
- При сбоях output contract агент перезапускается по recovery-лестнице с лимитом попыток из настроек (`max_qa_iterations`).
- При исчерпании recovery loop завершает задачу с явной диагностикой ошибки, а не подменяет результат шаблонным кодом.

## Риски и зависимости

- Риск: неуправляемая длительность и стоимость.
- Риск: непрозрачный "черный ящик" для пользователя.
- Зависимость: зрелая оркестрация, логирование, авторежим.

## Критерий готовности для демо

- Пользователь видит этапы автопилота и может остановить процесс.
- Есть минимум один успешный сценарий "с нуля до результата".
- При неуспехе система корректно завершает работу и отдает промежуточный результат.
